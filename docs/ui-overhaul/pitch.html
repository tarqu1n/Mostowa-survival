01 · Where it hurts

What the current UI gets wrong

Pulled from an audit of src/scenes/hud/* and src/ui/*. None of this is a knock on the code — it's a HUD that grew widget-by-widget and never got a layout system. Six things stand out.

Readability

8–11px monospace, everywhere

HP/food bars at 8px, hints at 8–9px, zoom buttons 24×24. Well below the ~44px / 16px-on-screen a phone needs. All text, no icons.

Layout

Hard-coded pixels, no reflow

Every widget computes x/y off fixed 360×640 constants. No safe-area insets for notches, no landscape, no adapting to aspect ratio beyond letterboxing.

Clutter

Corners stack by magic numbers

BUILD→CANCEL→ITEMS→DEMOLISH are manually offset; adding a button means re-tuning its neighbours. A dead, dimmed SPELL slot already sits on screen.

System

No design tokens

One font size, a colour set, and every gap/pad/size is a local literal. "Retune the whole look" isn't really possible today.

Reach

Controls scattered top-to-bottom

Primary actions live in the top-right red zone; the thumb-friendly bottom band is mostly unused outside combat. Modal panels open dead-centre over the action.

Interaction

Inventory is look-only

The slot grid can't select, drag, or equip (deferred). Panels each reinvent open/close, and none has a consistent close affordance.

02 · The field

What good phone survival UIs do

Distilled from Don't Starve: Pocket Edition, Kingdom Two Crowns, Last Day on Earth, The Long Dark, Whiteout Survival, and the mobile-UX literature. Ten principles that shaped every design below.

  1. 01Two layouts, not one rotation.Portrait = one thumb from a bottom corner. Landscape = move-left / act-right like a gamepad. Offer a left/right-hand mirror.
  2. 02Interactive lives in the bottom 30–40%.On a modern 6.5″ phone only the bottom half is comfortably one-hand reachable. Passive info (time, meters) goes up top.
  3. 03≥44–48px hit targets, decoupled from art.A 24px pixel-art icon carries an invisible 44px hit rectangle. This is the single biggest lever for a pixel game.
  4. 04World and UI are separate layers.World at integer nearest-neighbour zoom; UI baked crisp at a device scale so buttons stay tap-sized whatever the world zoom.
  5. 05Respect the notch.Full-bleed world behind the Dynamic Island; inset all controls to the safe rectangle. This is free in CSS, painful in hand-placed Phaser.
  6. 06Meters show state + trend, not numbers.Circular icon-meters. Persist only health, hunger, day/night. Fade everything else in on change.
  7. 07The day/night dial is the spine.Calm by day; at dusk surface "night is coming"; at night promote wave number + enemies-left to prominent overlay.
  8. 08Build = ghost + snap + confirm.Never place on raw touch-up (the finger hides the cell). Ghost snaps green/red → nudge → ✔/⟳/cancel → "place again" for wall runs.
  9. 09Combat = auto-target + context action.An "attack nearest" button with the target outlined beats tap-the-enemy. Surface weapon-swap only when you carry more than one.
  10. 10Acknowledge every tap; reserve red.Micro-scale/flash on press, juice the key moments (floaty "+5 wood", shake, particles), keep red exclusively for danger.

04 · The build call

Author the menus in HTML, not Phaser

Your instinct is right, and there's a strong precedent: the Map Builder already runs React + Tailwind v4 + shadcn/ui, deliberately kept out of the game bundle. The recommendation is a DOM overlay for the HUD and menus, with the world and in-world markers staying in Phaser.

Recommendation

Hybrid: a DOM/React HUD layer over the Phaser canvas

Nearly every pain point in §01 is something the browser solves for free and Phaser makes you hand-roll. Safe-area insets, responsive reflow, a real token system, crisp accessible text, portrait/landscape layouts — all native to CSS. And the HUD hit-test hack (hudHitTest gating world taps) largely disappears: DOM elements capture their own events; the canvas only sees what falls through.

DOM / React owns

  • HUD bars, meters, day/night dial, resource counts
  • Build tray, Pack & Craft sheets, inspect card, companion menu, main/pause menus
  • All buttons, drawers, modals — reusing the editor's shadcn primitives & Tailwind @theme tokens
  • Safe-area insets, responsive layout, focus/accessibility

Phaser keeps

  • The world, camera, entities, lighting — untouched
  • In-world markers that must live in world space: build ghost + grid snap, target outline, floating combat/gather text, monster HP bars, queue markers
  • Gesture mechanics on the canvas (tap / long-press paint / pan / pinch)

Costs to plan for

  • A thin event bridge: today's game.events bus fans out to React state (and back for intents). This is small and testable.
  • Coordinate mapping for the few DOM→world actions (place point, tap-to-eat targeting a world entity).
  • Tailwind now loads on the game page — scoped, and the editor proves the pipeline.

Why not stay in Phaser

  • You'd re-implement flexbox, safe-areas, and a token system by hand — the exact debt §01 is made of.
  • You'd maintain two UI idioms (editor DOM vs game Phaser) forever instead of one.

The three designs below are drawn as HTML for exactly this reason — what you tap through is the target stack. All three are achievable in either engine; the layout systems they need (reflow, insets, drawers) are what make DOM the easier author.

05 · The pitch

Three directions to choose from

Same game, same control set, three philosophies — from most minimal to most gestural. Each phone below is live: tap the chips under it to move through Scavenge · Build · Fight · Night and open the menus. Every control from the audit checklist has a home in all three.

Direction A

Emberlight

"Trust the world. Show almost nothing."

Diegetic-first. The screen is nearly all game. Meters are small glanceable rings that fade when calm; the day/night dial is the one permanent fixture. Build and menus rise as bottom sheets, then get out of the way. The firelight itself is the HUD — it dims as fuel drops.

Lineage: Kingdom Two Crowns · The Long Dark

Strengths
  • Most immersive, least clutter
  • Gorgeous on a small screen
  • Fewest tap targets to size
Risks
  • Can hide decision info (K2C's known flaw)
  • Discoverability leans on the player
  • Needs strong diegetic feedback to work

Where every control lives in Emberlight
Health / hungerTwo small rings, top-left, fade to 30% when full & calm; pulse red/amber at threshold. Tap hunger ring → eat list.
Day / night · waveSun-moon dial, top-centre, always on. At dusk it reddens; at night a slim wave/enemies-left strip unfurls beneath it.
ResourcesDiegetic — wood/rock counts appear only in Build mode, inside the sheet. Fire fuel = campfire brightness in-world.
BuildSingle ember Build FAB, bottom-right. Opens a bottom sheet of buildables → ghost+snap → floating ✔ ⟳ ✕ near the thumb.
CombatNothing until threat. Then invisible move-anywhere joystick (left) + one Attack orb (right, auto-target). Bow/weapon orb appears only if carried.
Pack / CraftSwipe up from bottom edge → tabbed sheet (Pack · Craft · Status). No permanent buttons.
Inspect / commandLong-press any entity → info card slides from bottom. Tap ground to move/order as today.
CompanionTap NPC → same bottom sheet, Day/Night posture rows.
Direction B · recommended starting point

Field Kit

"One bar that becomes whatever the moment needs."

A persistent bottom command bar that morphs by mode — Scavenge shows gather/build/pack; Build swaps to the buildable tray + confirm cluster; Fight swaps to move + attack. Meters sit top-left, the day/night dial top-centre. Pack and Craft are tabbed drawers. The pragmatic, legible, discoverable middle — closest to Don't Starve Pocket.

Lineage: Don't Starve: Pocket Edition · Last Day on Earth

Strengths
  • Everything discoverable & labelled
  • One consistent thumb zone
  • Scales cleanly to new tools/verbs
Risks
  • Bottom bar costs ~15% of screen
  • Least "novel"-feeling of the three
  • Mode morph needs clear transitions

Where every control lives in Field Kit
Health / hungerLabelled rings top-left, always visible with a small trend tick. Fire ring joins them when a hearth exists.
Day / night · waveDial top-centre + "Day N" label; NIGHT WAVE banner drops in under it at night with enemies-left count.
ResourcesWood/rock chips top-right, compact; expand in Build mode.
Command bar (bottom)Scavenge: Build · Pack · Craft · Status + Cancel-when-busy. Build: buildable tray + Rotate/Confirm/Cancel. Fight: move pad + Attack/Bow.
BuildBar becomes a horizontal buildable tray (cost + affordability dim) → ghost+snap → Rotate/✔/✕ on the bar → "place again" toggle.
CombatBar becomes move joystick (left) + Attack/Bow cluster (right). Auto-surfaces at threat; manual COMBAT toggle still available.

Direction C

Twin Grip

"Two thumbs, two wheels, whole screen."

Console-style, anchored to both bottom corners and built portrait and landscape from day one. Left corner = movement ring; right corner = a radial action wheel that pops under the thumb — its petals change by context (gather/build/attack). Build uses a radial category wheel too. The boldest, most gestural, most screen-forward — and the most to learn.

Lineage: console twin-stick · radial-menu action games

Strengths
  • Maximises world visibility
  • Landscape-native, fast in the hand
  • Feels premium / distinctive
Risks
  • Highest learning curve
  • Radials hide labels until opened
  • Hardest to build & tune

06 · Deciding

How I'd take this forward

My recommendation: build Field Kit (B) as the base — it's the most legible and discoverable, and it's the safest first overhaul — but steal Emberlight's contextual fade and diegetic fire-meter, and keep Twin Grip's corner anchoring in mind so a landscape mode is a layout swap later, not a rewrite. All three sit on the same DOM-overlay stack (§04), so the engine decision is independent of which look wins.

Once you pick a direction, the natural next step is a plan-feature pass: turn the chosen design into a staged implementation plan (event bridge → token system → HUD layer → per-mode surfaces), critique it, then execute. A few open questions worth settling first:

Q1Which direction — A, B, C, or a specific blend?
Q2Portrait-only for the first pass, or portrait + landscape from the start?
Q3DOM overlay confirmed? It's the recommendation, but it's a real architectural commitment worth your explicit yes.
Q4Pixel font or clean font for HUD text — full pixel identity, or pixel headers + clean readable body/numbers?
